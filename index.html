Message Yourself

01The process

what I tried, in order, and where each attempt landed.

CPT on Qwen 4B

I started with continued pre-training on Qwen 4B — a full dump of my iMessage history as one long text blob. The vocab started to feel familiar, but the hallucinations were too extreme to ignore: it would answer simple questions with paragraphs of invented detail.

More data: a Slack export

I figured maybe it just needed more volume, so I added a separate export of years of messages from a community Slack. Same story — fluent enough, still hallucinating.

Switched to SFT

Moved from pre-training to supervised fine-tuning: format the data as incoming messages → my reply, so the model learns to respond, not just continue text. Results got noticeably better.

Increasing context

Giving it the past 15 messages as context for the next one helped it produce more realistic-sounding results that made sense in the conversation.

Scaled up to gpt-oss-120b

Swapping the base model from Qwen 4B to gpt-oss-120b was the step that pushed quality over the line. SFT on top of a bigger base kept fluency intact while the adapter picked up the texture of how I text.

Tuning LR and LoRA rank

From there it was knob-twiddling. Two levers — learning rate and LoRA rank — and I couldn't tell the runs apart by eye. So I built an eval.

02An eval, then an agent

how I went from "can't tell these apart" to a config that actually worked.

I wrote an eval where Claude plays a friend in four scripted text-message conversations — a hike invite, a confide, a café follow-up, a podcast scheduling — and then grades the transcript on six rubrics: Realistic, Kind, Casual, Concise, No-Repeat, Natural.

First pass: nine runs spread across a reasonable band of learning rates and ranks. Best mean score was 3.75/5. Not bad, not great, and the differences between runs were small enough that picking the next combo by hand felt arbitrary.

So I gave Claude the wheel. It could queue training runs, read the eval scores as they landed, and decide what to try next. It ran for hours — I went to the grocery store — and eventually converged on lr=3e-5, rank=8 with a mean of 4.75/5.

04What it sounds like

notes from the best run.

The winning configuration — lr=3e-5, rank=8 — scores a 4.75/5 mean rubric. What that means in practice: the model stops leaking its own reasoning tags (that analysisassistantfinal nonsense you see in the base model), and it replies in one or two short lines like a real friend texting from the car.

It is not me. It's a decent impression. If you ask it something factual about my life it will confidently invent an answer. But for the shape and rhythm of a text thread — the kind, casual, concise signal — it's the closest I've gotten.

05Run it yourself

five commands. your raw messages stay on your laptop — only the LoRA weights travel.

# clone
git clone https://github.com/SerenityUX/message-yourself
cd message-yourself/CLI

# export tool for ~/Library/Messages/chat.db
brew install imessage-exporter

# venv + deps (system python3 is often 3.9; you want 3.12)
/opt/homebrew/bin/python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# keys — only Tinker sees adapter gradients, never your raw messages
cp .env.example .env   # fill TINKER_API_KEY, OPEN_ROUTER_API_KEY

# export iMessage → train pairs → SFT → eval → chat
python3 cli.py

Message export happens locally from ~/Library/Messages/chat.db. PEFT gradient computation happens on Tinker. Your raw message bodies never leave your machine — only the LoRA adapter weights do.
